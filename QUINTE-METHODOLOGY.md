# QueryForge 开发中的 AI 多智能体对抗审查体系

## 为什么用多智能体而不是单个 AI？

单个 AI 写代码有个致命问题：它不知道自己不知道什么。一个模型的盲区会直接变成产品的盲区。

QueryForge 采用了五方对抗审查协议（QUINTE），5 个独立 AI 智能体同时分析同一问题，互相挑刺，最终由人类+独立审计方做裁决。这不是装饰——每一轮都挖出了单个 AI 绝对找不到的 bug。

## 成本与 ROI

整套 5 方对抗审查体系的运营成本仅为小米 MiMo China Token Plan 的 API 调用费用：

- 底层模型：MiMo v2.5 Pro（小米自研）
- 5 个智能体共享同一模型，通过不同 CLI 工具实现独立性
- 单轮 R1（5 方并行）约消耗 50-80K tokens
- 单轮 R2（5 方交叉审查）约消耗 80-120K tokens（需读取全部 R1 输出）
- 完整三轮审查（R1+R2+R3）总消耗约 200-300K tokens
- 估算单次完整审查成本：约 ¥2-5（按小米 token plan 计价）

对比传统人工 Code Review：5 名工程师 × 2 小时 × ¥500/小时 = ¥5,000。
AI 多智能体审查的成本约为人工的 0.1%，且可在 10 分钟内完成。

这意味着即使在黑客松的 72 小时内进行 3-4 轮完整审查，总成本也不超过 ¥20。

## 对抗审查流程

每轮审查分三个阶段：

**R1 — 独立分析**：5 个智能体各自独立审查代码/方案，互不通信，避免群体思维。每个智能体用自己的工具链读代码、跑测试、写分析。

**R2 — 交叉审查**：5 个智能体匿名审查其他 4 个的 R1 输出（用 Participant A/B/C/D/E 代号），标记共识、分歧和盲区。如果被其他人的证据说服，必须声明立场变更及原因。

**R3 — 双路裁决**：人类开发者 + 独立审计方（未参与 R1/R2 的 Codex GPT-5.5）共同审查全部证据，产出包含残差闭环账本的最终裁决。

## 实际审查轮次与成果

### 第一轮：代码审计
- 5 方独立审查全部 10 个源文件
- 发现 5 个 P0 级缺陷（MetricSidebar 保存流程断裂、LLM 无超时、无离线 fallback、图表标题共享 bug、DB 连接不一致）
- R2 交叉审查确认 5/5 共识
- R3 裁决：CONDITIONAL PASS，全部 P0 修复后再放行

### 第二轮：技术方向决策
- 辩论是否保留 Next.js、部署方案、UI 风格
- R2 关键纠正：better-sqlite3 与 Vercel serverless 不兼容 → 改用 Railway
- 5 方否决了同事的单 HTML 方案（零 AI 深度）
- 最终裁决：Keep Next.js + Railway + 商务风 UI

### 第三轮：打磨策略审计
- 6 个维度：得分最大化、创新叙事、UI 优先级、Demo 流程、技术债分诊、部署决策
- 产出具体的时间分配方案和行动清单

## 这套体系的价值

1. **消除单点盲区**：5 个独立智能体的交集比任何单个都可靠
2. **强制证据链**：每个结论必须引用具体文件、行号、命令输出
3. **保留残差**：未解决的问题不会被抹平，而是记录为待处理项
4. **可审计**：所有 R1/R2/R3 输出文件完整保留，评委可逐行追溯

## 技术架构

```
用户提问 → Hermes Agent（宿主调度）
  ├── R1: 5 个 CLI 智能体并行分析（独立进程，互不通信）
  ├── R2: 5 个 CLI 智能体交叉审查（匿名代号）
  └── R3: 人类 + 独立审计方双路裁决
       └── 残差闭环账本（JSON 格式，每项有处置状态）
```

所有智能体通过独立的终端进程调度，不使用脚本包装或合并进程，确保审查的独立性和可追溯性。
