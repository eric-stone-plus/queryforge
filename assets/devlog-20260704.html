六、Round 4.5 — 架构审查（Codex）

三层架构设计
  • 受控语义层 — 分析师定义指标 → 系统匹配 → SQL 生成。不靠 LLM 猜表名
  • 验证式 Agent 循环 — SQL → AST 验证 → 只读执行 → 结果分析 → 可视化。自纠正
  • 企业数据平面 — Schema-only 模型暴露、只读数据库、AST 级 SQL 安全校验、审计日志路线图
对话质量升级

系统提示词从 "brief Chinese explanation" 升级为要求完整的分析师报告：

  • 数据结论：用数字说话，给出关键指标
  • 趋势/对比：与什么比较，是高是低
  • 业务建议：基于数据给出 1-2 条可执行建议
  • 数据局限性：说明任何需要注意的数据特点
数据合规方案
  • Schema-only 模型暴露 — LLM 只看到表名和列名，永远不接触原始数据
  • AST 级 SQL 验证 — node-sql-parser 解析语法树，只允许 SELECT
  • 只读数据库连接 — 数据库层面 readonly 模式
  • 自动 LIMIT 注入 — 防止全表扫描
  • 生产路线图 — 行级安全（RLS）、PII 脱敏、审计日志、SSO/SAML

十、关键教训

开发经验
  • 5/5 一致 = 必修 — 所有审查方都发现的问题，置信度最高，必须修复
  • 分歧 = 信号 — 不同意见不是噪声，而是需要人类判断的信号
  • 排练 > 代码 — 演示流程比功能完善更重要（5 方一致）
  • 真实数据 > 精心模拟 — Olist 的 3.1% 复购率比 faker 的 100% 更有说服力
  • 教训必须写进源头 — memory 不够，必须同步到 SKILL.md + SOUL.md
