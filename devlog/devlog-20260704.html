产品定位

从通用 Text-to-SQL demo 收敛到电商经营工具包

QueryForge 的最终定位是：让电商经营者获得专业数据分析能力。它不是通用聊天工具，也不是重型企业 BI，而是面向跨境电商小团队、平台卖家、外贸 SOHO、广交会参展商和小型贸易公司的本地优先经营工具包。

当前交付的是工具包里的第一个工具：订单与经营分析 workbench。真实用户通常是自己生意的老板或核心运营，分析的是自己的订单、商品、地区、渠道和复购数据。这个前提让产品更适合采用本地运行、用户自带模型 key、闭源预编译软件包分发的路径。

开发时间线

7/3 晚 — 命题发布
选定方向。最初从自然语言查询数据库切入，随后明确比赛需要展示商业价值，不应停留在“能生成 SQL”的技术 demo。
7/4 上午 — 基础产品完成
完成第一条分析链路。自然语言提问、SQL 生成、只读校验、SQLite 查询、图表和经营建议形成闭环，并补上客户端超时、错误兜底和缓存答案。
7/4 下午 — 数据切换
替换为 Kaggle Olist 真实数据。本地 SQLite demo 从旧模拟数据切换到 99,441 笔 Olist 巴西电商订单，形成地区、品类、渠道、客单价和复购分析的真实案例。
7/4 晚 — 产品路径调整
Railway 降级为 public QR demo。托管后端更新不稳定，不适合作为真实产品边界。Railway 保留给评委手机扫码，使用公开 Olist 数据、预设问题和缓存答案；真实产品路径切换到 macOS 本地应用。
7/5 — 桌面端打磨
完整本地应用路线。当前 demo 产物是 macOS QueryForge.app，打包本地 Next server、SQLite 数据库和 Node runtime。产品形态不限定 macOS，后续可以提供 Windows/macOS 预编译包。Settings 支持用户自助选择模型服务、填写 API key/token，并在高级设置调整 URL、backend、model 和 token 预算。
7/5 — 演示路径确认
现场 BYOK 演示。默认状态必须显示未配置；用户现场生成 DeepSeek key，打开桌面端，选择 DeepSeek，填写 key，确认 model 为 deepseek-v4-pro，保存并测试后直接自然提问。

关键产品决策

为什么不是直接问通用大模型
  • 通用模型默认没有本机数据库执行权，不知道用户自己的表结构和指标口径。
  • 通用模型不会自动保留 SQL、查询结果、指标口径和 token 用量记录。
  • QueryForge 的价值是 harness：schema 约束、只读 SQL、执行结果 grounding、图表、保存指标和预算管理。
为什么适合小微电商老板
  • 数据通常属于自己的经营数据，合规边界比大型企业跨部门数据治理更简单。
  • 不需要先采购 BI、建数据仓库或雇专职分析师。
  • 预编译软件包/买断首发更接近小老板的决策方式；模型 token 由用户自己的 provider 账户承担。
  • 广交会地推、卖家微信群、培训社群、服务商渠道和官网售卖都适合这种低客单价软件验证。

数据与指标

Olist demo case

Olist 是公开可复现的 demo case，不代表产品只能分析巴西电商。它用于证明 QueryForge 能处理真实订单结构和真实业务分布。

  • 99,441 订单 · 96,096 用户 · 32,951 商品 · 74 品类
  • 总营收 R$1,601 万 · 平均客单价 R$161 · 完成率 97% · 退款率 1.2%
  • 5 大地区均有订单：Sudeste、Sul、Nordeste、Centro-Oeste、Norte
  • 完成订单口径：Sudeste 66,200 单、Sul 13,814、Nordeste 9,044、Centro-Oeste 5,624、Norte 1,796
  • Norte/Nordeste 客单价较高，Sudeste 是规模核心市场

模型配置与隐私

BYOK 桌面端
  • 默认不读取仓库里的 .env.local，也不假装“模型已配置”。
  • 普通 Settings 只保留模型服务选择和 API key/token。
  • 高级设置提供 API URL、backend、model、timeout、temperature 和 token budget。
  • DeepSeek preset 默认使用 https://api.deepseek.com/v1 和 deepseek-v4-pro，便于现场演示。
  • 当前本地 key 存在用户本机配置文件并设置 0600 权限；商业发行版应迁移到操作系统凭据管理能力（macOS Keychain / Windows Credential Manager）。
公开 demo 边界
  • Railway 页面是手机/二维码演示，不收集访客 API key。
  • 公开页面使用 Olist demo case 和缓存答案，保证扫码体验稳定。
  • 真实模型调用、token 预算和用户数据连接只在本地桌面端发生。

商用输出与知识产权边界

主流模型服务的通常边界

本段不是法律意见，而是产品材料中的合规边界说明。公开条款显示，主流模型 provider 通常会定义输入/输出权利、用户责任和限制用途：OpenAI 条款写明用户保留 Input 权利并在适用法律允许范围内拥有 Output；Anthropic 公开商业条款说明客户可保留输出权利；DeepSeek 与 Kimi/Moonshot 条款强调用户需要对输入、输出和授权负责。

因此 QueryForge 的合规叙述应当是：工具接入用户自己的模型 provider 账户，服务于用户自己的经营数据分析；QueryForge 不转售模型能力，也不承诺替用户消除第三方数据、平台条款、个人信息保护或知识产权责任。

对目标用户的实际含义
  • 老板分析自己的订单和商品数据，通常具备处理这些经营数据的业务合理性。
  • 如果数据包含客户个人信息、平台限制字段、第三方图片/文案或他人商业秘密，仍需先确认处理和上传给模型服务的权利。
  • 输出建议应作为经营分析参考，关键决策仍需要结合毛利、库存、物流、广告成本和人工判断。
