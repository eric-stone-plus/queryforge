## Project Overview

QueryForge is a Text-to-SQL data analysis agent. Business users ask questions in natural language, the system generates SQL via Kimi K2.7 Code LLM, executes against a SQLite database, and returns visualized results with anomaly analysis and decision suggestions.

Tech stack: Next.js 14, Tailwind, Recharts, better-sqlite3, Kimi K2.7 Code (Vercel AI SDK), Railway deployment, macOS desktop (SwiftUI).

Key files:
- src/lib/agent.ts — AI reasoning + self-correction loop (187 lines)
- src/app/api/chat/route.ts — SSE streaming API (47 lines)
- src/app/page.tsx — Main UI with hardcoded KPIs + static chart data (319 lines)
- src/components/ChatPanel.tsx — Chat interface + chart rendering (341 lines)
- src/components/Dashboard.tsx — EXISTS BUT NOT WIRED INTO page.tsx
- src/components/MetricSidebar.tsx — Analyst preset metrics
- src/lib/db.ts — SQLite readonly connection (16 lines)
- src/lib/demo-cache.ts — 4 pre-cached demo results (31 lines)
- scripts/seed.ts — Generates FAKE data with faker.js (373 lines)
- data/ecommerce.db — 10K orders, 25K order_items, 500 products, 1000 users, 8 regions

## Current Data: 100% Fabricated

The entire database is generated by faker.js with seed(20260704). This means:
- All product names are fake Chinese brand+product combos (e.g. "云启瑜伽垫", "海棠机械键盘")
- All user names are faker-generated
- All order patterns are random distributions
- The "100% repurchase rate" and "人均10单" KPIs are artifacts of 1000 users × 10 orders each
- No real-world data patterns, seasonality, or business logic

## Competitor Landscape (Research Findings)

### Major Competitors:

1. **Vanna.ai** (Open Source, 12K+ GitHub stars)
   - RAG-based: trains on custom SQL examples for higher accuracy
   - Supports multiple LLMs and databases
   - Has Flask/Streamlit UI, REST API
   - Weakness: no built-in anomaly detection or decision suggestions

2. **BlazeSQL** (Commercial SaaS)
   - Claims 70-85% accuracy on clean data out-of-box
   - Natural language → SQL → visualization pipeline
   - Production-grade, multi-database support
   - Weakness: closed-source, expensive

3. **AskYourDatabase** (Commercial SaaS)
   - Natural language → interactive dashboards
   - "No SQL required" positioning
   - Real-time dashboard generation
   - Weakness: limited customization

4. **Wren AI** (Open Source, Enterprise)
   - Semantic layer / metadata management
   - GenBI (Generative BI) approach
   - Enterprise features: multi-tenancy, RBAC
   - Weakness: complex setup

5. **SQL Chat** (Open Source)
   - Chat-based SQL generation
   - Simple, clean UI
   - Weakness: basic features, no analytics

6. **Waii.ai** (Enterprise API)
   - Enterprise Text-to-SQL API
   - Multiple visualization frameworks
   - Weakness: API-only, no self-service UI

7. **Oracle Select AI / Salesforce Horizon Agent**
   - Big-enterprise internal tools
   - Not direct competitors but set expectations

### Key Differentiators in the Market:

- RAG-based training (Vanna): learns from your actual SQL patterns
- Semantic layer (Wren AI): understands business metrics, not just schema
- Multi-database support (most competitors): PostgreSQL, MySQL, BigQuery, etc.
- Self-correction with confidence scoring
- Collaborative features (sharing queries, dashboards)
- Data governance / RBAC

### Q3: Data Source Evaluation
Should we replace the faker-generated data with a real public dataset? Evaluate these options:

Option A: Keep faker data as-is
- Pro: No work needed, schema already matches
- Con: Looks fake, "100% repurchase" is embarrassing, no real patterns

Option B: Olist Brazilian E-commerce Dataset (Kaggle, 100K orders)
- Pro: REAL data from actual Brazilian e-commerce platform, 100K orders, rich schema (orders, payments, reviews, products, sellers, geolocation), widely recognized in data community
- Con: Need to transform CSV→SQLite, adapt schema, update queries and system prompt, different column names

Option C: Chinook Music Store Database
- Pro: Already SQLite, well-known, clean schema, 11K+ rows
- Con: Music store domain different from e-commerce narrative, smaller dataset

Option D: Google Analytics Ecommerce Demo Dataset
- Pro: Real Google data, BigQuery-native
- Con: BigQuery not SQLite, would need conversion

Option E: Generate better faker data with realistic patterns
- Pro: Keep schema, add seasonality, realistic distributions, fix broken KPIs
- Con: Still synthetic, but much more believable

Recommend the best option considering: time available (Demo Day is tomorrow), scoring impact, data credibility, and implementation effort.
