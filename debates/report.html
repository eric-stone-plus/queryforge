一、审查概览

QUINTE 协议

QUINTE 是一个结构化残差暴露协议。每轮审查由 5 个独立 AI 智能体并行执行，交叉审查后由人类裁决。核心思想：不追求共识，而是暴露矛盾、遗漏和证据缺口。

  • R1（独立审查） — 5 方各自独立分析，互不参考
  • R2（交叉审查） — 5 方互相审查 R1 输出，标记共识、分歧和遗漏
  • R3（裁决） — 人类 + 独立审计方综合 R1/R2 证据，产出行动清单

三、Round 2 — 方向决策

核心发现
  • 创新性短板：Text2SQL 不新颖（Vanna.ai 12K stars，BlazeSQL，Wren AI 都在做）
  • 差异化定位：5 方各自提出方案，裁决合并为"分析师定义一次，业务团队永久使用"
  • MetricSidebar 是核心差异化：竞品没有"分析师预设指标库"功能
  • 自纠正循环是真实创新：SQL 出错 → 错误反馈 → 自动修正 → 重试，竞品不做这个

数据策略决策

5 方初始推荐 Option E（修 faker），但执行期间完成了 Olist 迁移（实际 5 分钟 vs 预估 4-6 小时）。关键教训：QUINTE 的风险评估帮助识别"看起来高风险但实际可控"的任务。

迁移结果：

  • faker: 10K 订单 · 500 商品 · 1000 用户 · 100% 复购率
  • Olist: 99K 订单 · 33K 商品 · 96K 用户 · 3.1% 复购率

六、Round 4.5 — 架构审查（Codex）

三层架构设计
  • 受控语义层 — 分析师定义指标 → 系统匹配 → SQL 生成。不靠 LLM 猜表名
  • 验证式 Agent 循环 — SQL → AST 验证 → 只读执行 → 结果分析 → 可视化。自纠正
  • 企业数据平面 — Schema-only 模型暴露、只读数据库、AST 级 SQL 安全校验、审计日志路线图
对话质量升级

系统提示词从 "brief Chinese explanation" 升级为要求完整的分析师报告：

  • 数据结论：用数字说话，给出关键指标
  • 趋势/对比：与什么比较，是高是低
  • 业务建议：基于数据给出 1-2 条可执行建议
  • 数据局限性：说明任何需要注意的数据特点
数据合规方案
  • Schema-only 模型暴露 — LLM 只看到表名和列名，永远不接触原始数据
  • AST 级 SQL 验证 — node-sql-parser 解析语法树，只允许 SELECT
  • 只读数据库连接 — 数据库层面 readonly 模式
  • 自动 LIMIT 注入 — 防止全表扫描
  • 生产路线图 — 行级安全（RLS）、PII 脱敏、审计日志、SSO/SAML

十、关键教训

开发经验
  • 5/5 一致 = 必修 — 所有审查方都发现的问题，置信度最高，必须修复
  • 分歧 = 信号 — 不同意见不是噪声，而是需要人类判断的信号
  • 排练 > 代码 — 演示流程比功能完善更重要（5 方一致）
  • 真实数据 > 精心模拟 — Olist 的 3.1% 复购率比 faker 的 100% 更有说服力
  • 教训必须写进源头 — memory 不够，必须同步到 SKILL.md + SOUL.md
